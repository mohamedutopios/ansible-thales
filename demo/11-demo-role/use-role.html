1. Préparation du rôle

Assurez-vous que :

  • Le rôle est bien créé (vous pouvez vérifier en naviguant dans son répertoire).
  • Le rôle se trouve dans un des chemins définis dans ansible.cfg (section roles_path). Par défaut, les rôles sont placés dans le répertoire roles du projet.

Si votre rôle "webserver" se trouve dans un répertoire non standard, configurez roles_path dans votre fichier ansible.cfg :

[defaults]
roles_path = ./roles:/etc/ansible/roles

2. Utilisation dans un playbook

a. Créer un fichier playbook.yml

Ajoutez le rôle dans votre playbook sous la section roles :

---
- name: Déployer un serveur web
  hosts: webservers
  become: yes  # Permet d'exécuter les tâches avec des privilèges élevés (sudo)

  roles:
    - webserver

b. Exemple d'inventaire

Définissez les hôtes cibles dans un fichier inventory :

[webservers]
192.168.1.10 ansible_user=centos ansible_ssh_private_key_file=~/.ssh/id_rsa
192.168.1.11 ansible_user=centos ansible_ssh_private_key_file=~/.ssh/id_rsa

3. Exécution du rôle avec Ansible

Lancez le playbook qui utilise le rôle avec la commande suivante :

ansible-playbook -i inventory playbook.yml

Résultat attendu :

  • Ansible applique toutes les tâches définies dans le rôle "webserver" sur les hôtes spécifiés dans inventory.
  • Les résultats de chaque tâche (succès, échec ou modification) seront affichés.

4. Gestion des variables dans le rôle

a. Variables par défaut (defaults/main.yml)

Si des variables sont définies dans defaults/main.yml du rôle, elles seront utilisées par défaut, sauf si elles sont surchargées.

Exemple dans defaults/main.yml du rôle :

---
nginx_port: 80

Vous pouvez surcharger cette variable dans le playbook :

---
- name: Déployer un serveur web avec un port personnalisé
  hosts: webservers
  become: yes

  vars:
    nginx_port: 8080  # Surcharge de la variable par défaut

  roles:
    - webserver

b. Variables par groupe ou hôte

Vous pouvez définir des variables spécifiques à un groupe ou un hôte dans l'inventaire :

[webservers]
192.168.1.10 ansible_user=centos ansible_ssh_private_key_file=~/.ssh/id_rsa nginx_port=8080
192.168.1.11 ansible_user=centos ansible_ssh_private_key_file=~/.ssh/id_rsa nginx_port=9090

5. Tester et valider le rôle

Avant d'utiliser le rôle en production, vous pouvez le tester dans un environnement local ou de test. Voici quelques options pour tester un rôle :

a. Tester le rôle seul

Créez un fichier test_playbook.yml pour tester uniquement le rôle :

---
- name: Tester le rôle webserver
  hosts: localhost
  roles:
    - webserver

Exécutez-le :

ansible-playbook -i localhost, test_playbook.yml

b. Mode simulation (dry-run)

Utilisez l'option --check pour voir ce qu'Ansible ferait, sans réellement appliquer les changements :

ansible-playbook -i inventory playbook.yml --check

6. Organisation des rôles dans un projet

Dans un projet réel, il est courant d'organiser les rôles dans un répertoire dédié. Voici un exemple d'organisation typique :

project/
├── ansible.cfg
├── inventory
├── playbook.yml
└── roles/
    └── webserver/
        ├── defaults/
        ├── files/
        ├── handlers/
        ├── meta/
        ├── tasks/
        ├── templates/
        └── vars/

Assurez-vous que le chemin vers le répertoire roles/ est défini dans ansible.cfg.

7. Utilisation avancée avec plusieurs rôles

Dans un playbook, vous pouvez utiliser plusieurs rôles. Par exemple :

---
- name: Déployer un serveur web avec base de données
  hosts: all
  become: yes

  roles:
    - webserver   # Configure Nginx
    - mysql       # Configure MySQL
